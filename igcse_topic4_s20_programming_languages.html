⌨️ Why Do Programming Languages Matter? ▼ expand
🤖 CPUs Only Understand Binary

Every program must ultimately become machine code (binary 1s and 0s) before the CPU can run it. Understanding how languages get translated to machine code is fundamental.

🔧 Assembly is Still Used

Assembly language is still used for device drivers, embedded systems and performance-critical code where direct hardware control and tiny program size matter.

⚡ Compiler vs Interpreter

This is a classic exam topic. You must know what each does, how they differ in the translation process, and the advantages and disadvantages of each approach.

📊 Language Levels

High-level languages (Python, Java, C#) are human-readable; low-level languages (assembly, machine code) are closer to the CPU's native instructions. Each has its own use case.

🛡️ Object Code & Protection

Compiled programs produce object code which can be distributed without revealing source code — protecting intellectual property. Interpreted programs require the source code to run.

🐛 Debugging Differences

Interpreters find errors line by line; compilers report all errors at once and won't produce output until the whole program is correct. Both have implications for development.

✏️ Key distinction: Machine code = binary, CPU executes directly. Assembly = mnemonics, translated by assembler. High-level = human-readable, translated by compiler OR interpreter.

20.1Low-Level Languages: Machine Code & Assembly
▼

A CPU can only carry out a small number of simple operations. The list of all possible commands a CPU knows is called its instruction set. Each instruction is given a unique binary code. The binary codes representing a program are called machine code.

SUBJECT VOCABULARY

Instruction set — the list of all possible commands a particular CPU knows how to carry out.

Machine code — the binary codes representing each of the instructions in the instruction set. The only language a CPU can directly execute.

Translator — a program that converts source code into machine code.

Source code — the text of the program that a programmer writes.

Assembler — a translator that converts assembly language mnemonics into machine code instructions for the CPU to carry out.

Assembly language — a low-level language written using mnemonics (e.g. ADD, LDR, STR).

Mnemonic — a short, memorable keyword representing a CPU instruction (e.g. ADD for addition, LDR for load register, STR for store, CMP for compare).

Low-level programming language — a programming language closely related to the CPU's machine code (machine code and assembly language).

🤖 Machine Code

Machine code looks like this — a string of binary digits:

01101001001001010101010111110101011001010110101010

Writing programs in machine code is completely impractical — it would take enormous effort to work out the correct binary sequence, and the program would only run on one specific type of CPU (since different CPUs have different instruction sets).

🔧 Assembly Language

Assembly language replaces those binary codes with short memorable mnemonics. For example, the instruction to add numbers (binary 01101001) is simply written as ADD in assembly.

Here is an example of assembly code:

LDR R3, [R1] ; Load value from memory address in R1 into R3 MQV R2, #0 ; Move value 0 into register R2 CMP R3, R0 ; Compare R3 with R0 MOVGT R2, #1 ; If R3 > R0, move 1 into R2

The assembler replaces each mnemonic with the appropriate binary machine code.

⚠️ Why Assembly is Difficult

Limited Instructions

Only the most basic operations exist. Every task must be built from the smallest steps. Some older CPUs could not even multiply.

Manual Data Management

No strings, integers or real numbers — just binary. The programmer must decide how to represent and where to store all data in memory.

Very Hard to Debug

Any bugs usually cause the machine to crash and require a reboot. There are no helpful error messages like in high-level languages.

✅ When Assembly IS Used

Despite the challenges, assembly is still used for:

✓
Hardware device drivers — need to directly control I/O hardware at the lowest level
✓
Embedded systems — need programs that are very small and very fast
✓
Performance-critical sections — where every nanosecond matters
✓
Direct hardware control — when you need to control the CPU's registers directly
✏️ Exam tip: Assembly language is a low-level language. Machine code is ALSO low-level. The key difference: machine code = binary; assembly = mnemonics translated by an assembler. Both are CPU-specific — programs written for one CPU type will not run on another.
20.2High-Level Languages
▼

High-level languages (such as Python, C#, Java, Visual Basic) are designed to be human-readable. They abstract away the detail of the hardware — a programmer uses meaningful commands like print("Hello") without needing to know anything about the CPU's registers or memory addresses.

# High-level Python — easy for humans to read for i in range(5): print("Hello, World!") # Compare with what machine code would look like: 01101001 00100101 01010101 01111101 01011001...
✅ Advantages of High-Level
✓
Human-readable — easier to write, read and maintain
✓
Portable — same source code runs on different hardware (via OS)
✓
Has data types, variables, loops, functions — faster to develop
✓
Easier to debug — meaningful error messages
❌ Compared to Assembly
✗
Less control over hardware — cannot directly access CPU registers
✗
Slightly less efficient — machine code generated by compiler may not be perfectly optimised
✗
Must be translated before the CPU can run it
✏️ Exam tip: "Portable" means the same source code can run on different types of hardware (as long as the correct translator is available). Assembly is NOT portable — it is specific to one CPU's instruction set.
20.3Compilers vs Interpreters — In Detail
▼
SUBJECT VOCABULARY

Compiler — a translator that converts high-level language source code into object code (often machine code) all at once. The source code is translated once and the saved object code can be run later without needing the compiler.

Interpreter — a translator that converts high-level language source code into machine code one line at a time, executing each line immediately. There is no saved object code — the source code must be translated every time the program runs.

Object code — the translated output from a compiler. Often machine code, sometimes an intermediate code. Saved as an executable file (.exe).

20.4The Assembler — Translating Assembly to Machine Code
▼

The assembler is the simplest type of translator. It converts assembly language programs (written with mnemonics) into machine code that the CPU can execute directly. Unlike a compiler, there is a one-to-one relationship — each assembly instruction becomes exactly one machine code instruction.

🔍 Click to zoom & download
💡 1:1 mapping: One mnemonic → one binary instruction. This is different from a compiler, which may convert one high-level statement into many machine code instructions (e.g. a Python for loop might produce dozens of machine code instructions).
✏️ Three translators summary: Assembler = assembly → machine code (1:1). Compiler = high-level → object code (whole program, saved). Interpreter = high-level → machine code (one line at a time, not saved).

Question 2 Medium4 marks

Describe the difference between a compiler and an interpreter. In your answer, refer to how each translator works and the output each produces.

✅ Mark Scheme
  • A compiler translates the whole program at once [1]
  • A compiler produces object code (saved machine code / .exe file) that can be run later without the compiler [1]
  • An interpreter translates and runs the source code one line at a time [1]
  • An interpreter produces no saved object code — the source code must be present and retranslated every time the program runs [1]
Question 3 Medium4 marks

Give two advantages of using a compiler and two advantages of using an interpreter when developing and distributing software.

✅ Mark Scheme

Compiler advantages (any 2):

  • The compiled object code runs faster because it has already been translated — no translation needed at runtime [1]
  • The object code can run on any similar computer without the compiler being installed [1]
  • Source code is protected — only the object code (machine code) is distributed, which is hard to reverse-engineer [1]

Interpreter advantages (any 2):

  • Errors are found line by line — the interpreter stops at the first error and reports it, making it easier to locate bugs [1]
  • Useful during development — can test individual lines without compiling the whole program [1]
  • More portable for scripting — the same source code runs on any system with the interpreter installed [1]
Question 4 Medium4 marks

Explain why assembly language is considered a 'low-level' language. Give two reasons why a programmer might still choose to write in assembly language rather than a high-level language.

✅ Mark Scheme
  • Assembly language is low-level because it is closely related to the CPU's machine code — each mnemonic corresponds directly to one machine code instruction [1]
  • It works at the level of the CPU hardware itself (registers, memory addresses) rather than abstracting away hardware detail like high-level languages [1]
  • Reason 1: Programs written in assembly can be very small and very fast — the programmer has full control over memory and the CPU [1]
  • Reason 2: Needed for direct hardware control — for example, writing device drivers or embedded system firmware where specific I/O operations must be performed [1]
Question 5 Hard6 marks
A games company develops a new 3D game. They compile their source code using a C++ compiler. A student is writing a school project in Python, which uses an interpreter.

Compare the use of a compiler for the games company's software with the use of an interpreter for the student's project. In your answer refer to: translation process, error detection, speed of execution, and distribution of the program.

✅ Mark Scheme (1 mark per valid comparative point, max 6)
  • Translation: The compiler translates the whole C++ program at once; the interpreter translates the Python code one line at a time [1]
  • Object code: The compiler produces object code (.exe) saved to disk; the interpreter produces no saved object code [1]
  • Error detection: The compiler reports all errors at once — the game cannot be compiled until all errors are fixed; the interpreter stops at each error line by line, making it easier for the student to debug [1]
  • Speed: The compiled game runs faster as it is already translated; the Python script runs slower as the interpreter translates each line at runtime [1]
  • Distribution: The games company can distribute the object code without exposing their source code, protecting their intellectual property; the student must share their Python source code for it to run [1]
  • Dependencies: The compiled game runs on any compatible computer without the compiler; the student's Python script requires Python (the interpreter) to be installed on any machine that runs it [1]

Question 7 Hard5 marks
A software developer has written a program in a high-level language. They are deciding whether to use a compiler or an interpreter to translate their program.

Discuss the advantages and disadvantages of each approach. Conclude with a recommendation for which the developer should use when: (a) testing/developing the program, and (b) distributing the finished product to customers.

✅ Mark Scheme

Interpreter during development (up to 2):

  • Finds errors line by line — easier to identify and fix bugs during development [1]
  • Can run partial code without having a complete, error-free program [1]

Compiler for distribution (up to 2):

  • Produces object code that runs faster — better user experience for customers [1]
  • Object code can run without the compiler installed — customers don't need to install the translator [1]
  • Source code is protected — customers only receive machine code which is hard to copy or reverse-engineer [1]

Conclusion (1 mark for both justified):

  • Use an interpreter during development/testing; use a compiler to produce the final product for distribution [1]
